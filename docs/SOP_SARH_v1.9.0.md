## 2. دليل المسارات وآلية الربط الآمن

### 2.1 المشكلة الأمنية

في الاستضافة المشتركة (Shared Hosting)، المجلد الذي يراه المتصفح هو `public_html/`. إذا وُضع كامل مشروع Laravel داخله، ستكون ملفات `.env` وقاعدة البيانات والكود المصدري **مكشوفة للعالم**. هذا خطر أمني كارثي.

### 2.2 الحل المعتمد: فصل المسارات

```
/home/u850419603/
├── sarh/                          ← 🔒 مشروع Laravel الكامل (خارج الويب)
│   ├── app/
│   ├── bootstrap/
│   ├── config/
│   ├── database/
│   ├── lang/
│   ├── resources/
│   ├── routes/
│   ├── storage/
│   ├── vendor/
│   ├── .env                       ← 🔒 بيانات حساسة (غير مكشوفة)
│   └── public/                    ← ✅ المجلد العام الوحيد
│       ├── build/                 ← أصول Vite المُجمّعة (CSS/JS)
│       ├── .htaccess
│       └── index.php              ← نقطة الدخول الأصلية
│
└── public_html/                   ← 🌐 ما يراه المتصفح
    └── → symlink → /home/u850419603/sarh/public
```

**النتيجة:** المتصفح يصل فقط إلى محتويات `public/` عبر الرابط الرمزي (symlink). جميع الملفات الحساسة تبقى **فوق** مجلد الويب العام.

### 2.3 إنشاء الرابط الرمزي

```bash
# إزالة المجلد القديم وإنشاء الرابط
rm -rf /home/u850419603/public_html
ln -s /home/u850419603/sarh/public /home/u850419603/public_html
```

**التحقق:**
```bash
ls -la /home/u850419603/public_html
# يجب أن يُظهر: public_html → /home/u850419603/sarh/public
```

### 2.4 ملف index.php — الجسر الحرج

ملف `index.php` هو **نقطة الدخول الوحيدة** لكل الطلبات. في بيئة Hostinger مع الوكيل العكسي (Reverse Proxy)، يجب استخدام **مسارات مطلقة** بدلاً من `__DIR__` النسبية:

```php
<?php
// filepath: public/index.php — الجسر بين الويب وقلب Laravel

use Illuminate\Http\Request;

define('LARAVEL_START', microtime(true));

// ── ملف الصيانة (Maintenance Mode) ──
$maintenance = '/home/u850419603/sarh/storage/framework/maintenance.php';
if (file_exists($maintenance)) {
    require $maintenance;
}

// ── تحميل Composer Autoloader ──
require '/home/u850419603/sarh/vendor/autoload.php';

// ── تحميل نواة Laravel ومعالجة الطلب ──
(require_once '/home/u850419603/sarh/bootstrap/app.php')
    ->handleRequest(Request::capture());
```

> **⚠️ تحذير حرج:** أمر `deploy.sh` ينسخ ملفات `public/` إلى `public_html/` ثم **يُعيد كتابة** `index.php` بالمسارات المطلقة. إذا تم تشغيل `php artisan optimize` بعد النسخ بدون إعادة الكتابة، يتعطل التطبيق فوراً.

### 2.5 حماية .htaccess

ملف `.htaccess` في `public/` يوجّه جميع الطلبات إلى `index.php` ويحمي من الوصول المباشر:

```apache
# توجيه كل الطلبات إلى index.php
RewriteEngine On
RewriteRule ^(.*)$ index.php [L,QSA]

# حماية الملفات الحساسة
<FilesMatch "^\.">
    Order allow,deny
    Deny from all
</FilesMatch>
```

### 2.6 الثقة بالوكيل العكسي (Trusted Proxies)

في `bootstrap/app.php`، تم تفعيل الثقة بجميع الوكلاء لأن Hostinger يستخدم وكيلاً عكسياً أمام Apache:

```php
$middleware->trustProxies(at: '*');
```

**بدون هذا السطر:**
- `$request->isSecure()` يُرجع `false` رغم أن الاتصال HTTPS
- رمز CSRF لا يتطابق → خطأ **419 Page Expired**
- الجلسات (Sessions) تتعطل

## 4. مصفوفة الصلاحيات الكاملة

### 4.1 نظام المستويات الأمنية (Security Levels)

نظام صرح يعتمد على **10 مستويات أمنية** مع حقلين حاسمين في جدول `users`:
- **`security_level`** (1–10): لا يمكن تغييره إلا عبر `setSecurityLevel()`
- **`is_super_admin`** (boolean): لا يمكن ترقيته إلا عبر `promoteToSuperAdmin()`

### 5.4 خطأ 419 — Page Expired (CSRF)

**السبب الأكثر شيوعاً في Hostinger:** الوكيل العكسي يُحوّل الطلبات من HTTPS إلى HTTP داخلياً. هذا يجعل Laravel يظن أن الجلسة غير آمنة.

**الحل (مُطبّق في v1.9.0):**
ملف `bootstrap/app.php` يحتوي على:
```php
$middleware->trustProxies(at: '*');
```

**إذا استمر الخطأ:**
```bash
# تنظيف الجلسات
php artisan session:table   # (إذا كانت الجلسات في قاعدة البيانات)
php artisan cache:clear
php artisan config:clear

# التحقق من إعدادات الجلسة في .env
grep SESSION .env
# يجب أن يكون: SESSION_DRIVER=file أو database
# يجب أن يكون: SESSION_SECURE_COOKIE=true (في الإنتاج مع HTTPS)
```

```
│                  طبقة الأحداث (Event-Driven)               │
│  BadgeAwarded → HandleBadgePoints (PerformanceAlert)      │
│  TrapTriggered → LogTrapInteraction (AuditLog)            │
│  AttendanceRecorded → (مُعَد للتوسع المستقبلي)           │
├──────────────────────────────────────────────────────────┤
│               طبقة المهام (Queue Jobs)                     │
│  ProcessAttendanceJob (async check-in, 30s timeout)       │
│  SendCircularJob (batch 100, 120s timeout)                │
│  RecalculateMonthlyAttendanceJob (monthly, scope: all)    │
├──────────────────────────────────────────────────────────┤
│              طبقة الاستثناءات (Exception Architecture)     │
│  BusinessException (رسالة مستخدم + سياق + كود HTTP)       │
│  OutOfGeofenceException (مسافة + نصف القطر المسموح)       │
│  bootstrap/app.php: reportable + renderable               │
├──────────────────────────────────────────────────────────┤
│              طبقة التعلم الآلي (ML Stub)                   │
│  ChurnPredictor: تأخر + غياب + نقاط → خطر المغادرة       │
├──────────────────────────────────────────────────────────┤
│              طبقة التوثيق التلقائي (API Docs)              │
│  Scramble v0.13.13 → /docs/api                            │
│  ApiDocsPage (Filament) → المستوى 7+                      │
└──────────────────────────────────────────────────────────┘
```

### 8.3 الأحداث والمستمعون (Events & Listeners)

| الحدث | المستمع | التأثير |
|-------|---------|---------|
| `BadgeAwarded` | `HandleBadgePoints` | إنشاء `PerformanceAlert` بنوع `badge_earned` مع تفاصيل الشارة والنقاط |
| `TrapTriggered` | `LogTrapInteraction` | تسجيل في `AuditLog` بنوع `trap.triggered` مع بيانات المصيدة ومستوى الخطر |
| `AttendanceRecorded` | *(مُعد للتوسع)* | يُطلق بعد حفظ أي `AttendanceLog` — بوابة لربط أنظمة مستقبلية |

**التسجيل في `AppServiceProvider`:**
```php
Event::listen(BadgeAwarded::class, HandleBadgePoints::class);
Event::listen(TrapTriggered::class, LogTrapInteraction::class);
```

### 8.4 المهام غير المتزامنة (Queue Jobs)

#### `ProcessAttendanceJob`
- **الغرض:** تسجيل حضور عبر الطابور (HTTP 202)
- **المدخلات:** User, latitude, longitude, ip, device
- **التسلسل:** فرع → جيوفنس → وردية → سجل حضور → تقييم → لقطة مالية → حدث `AttendanceRecorded`
- **timeout:** 30 ثانية | **tries:** 3
- **المسار:** `POST /attendance/queue-check-in`

#### `SendCircularJob`
- **الغرض:** إرسال تعميم لمئات الموظفين بدون تعطيل الطلب
- **timeout:** 120 ثانية | **tries:** 2
- **الآلية:** تقسيم المستخدمين إلى دفعات (100) + إنشاء `PerformanceAlert` لكل موظف + `sleep(1)` بين الدفعات

#### الجدولة (Schedule)
```
أول يوم من كل شهر الساعة 02:00 → RecalculateMonthlyAttendanceJob::forMonth()
أسبوعياً → queue:flush (تنظيف المهام الفاشلة)
```
