# الميثاق الوظيفي لنظام صرح — SARH Functional Charter

> **إصدار:** 4.0.0 | **تاريخ التحديث:** 2026-02-13
> **نطاق هذا المستند:** شرح كامل لكل وحدة في النظام وقيمتها التجارية للإدارة

---

## 1. نظرة عامة على النظام

**صرح الإتقان (SARH AL-ITQAN)** هو نظام ذكاء مالي وموارد بشرية مؤسسي عالي الأمان. الهدف الأساسي هو تمكين المدير من مراقبة كل شيء بشكل بصري بدون أي مجهود — "لوحة القيادة بدون جهد" (Zero-Effort Dashboard).

**المحور المالي الذكي:** كل دقيقة تأخير لها ثمن محسوب رياضياً من الراتب الأساسي للموظف. هذا يحول الحضور من "بيانات مجردة" إلى "خسائر مالية فعلية" يفهمها المدير فوراً.

---

## 2. الوحدات الرئيسية وقيمتها التجارية

### 2.1 إدارة الفروع والحضور الجغرافي (Branches & GPS Attendance)

**ماذا يفعل للمدير؟**

- يضمن أن الموظف **فعلياً في موقع العمل** وليس في مكان آخر
- يستخدم حدود جغرافية دقيقة بنصف قطر **17 متراً** حول مركز كل فرع
- يعتمد على صيغة **Haversine** لحساب المسافة بدقة

**المكونات:**

| الجدول        | الوظيفة                                                                                                                                        |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| `branches`        | بيانات الفروع مع الإحداثيات الجغرافية (خط الطول والعرض) ونصف قطر الحدود الجغرافية |
| `attendance_logs` | سجل كامل لكل تسجيل حضور بإحداثيات GPS والمسافة من المركز والحالة المالية                    |
| `shifts`          | جداول المناوبات مع فترة السماح                                                                                              |
| `user_shifts`     | تعيينات المناوبات للموظفين — كيان مستقل بصلاحية زمنية وتدقيق (v3.4)                                               |
| `holidays`        | العطل الرسمية لحساب أيام الغياب المبرر                                                                               |

**كيف يمنع الهدر المالي؟**

- كل تسجيل حضور **يحفظ المسافة الفعلية بالأمتار** — لا يمكن التلاعب
- إذا كان الموظف خارج الـ 17 متر، يُرفض التسجيل أو يُوثق كمخالفة

---

### 2.2 الذكاء المالي — محرك تكلفة الدقيقة (Financial Intelligence Engine)

**ماذا يفعل للمدير؟**

- يحول **كل دقيقة تأخير** إلى **مبلغ مالي** مفهوم
- يعرض **الخسائر المالية الإجمالية** حسب الفرع / القسم / الموظف / الفترة الزمنية
- يسمح بمقارنة **ميزانية الرواتب** مقابل **الخسائر الفعلية**

**المعادلة الأساسية:**

```
تكلفة_الدقيقة = الراتب_الأساسي ÷ أيام_العمل_الشهرية ÷ ساعات_العمل_اليومية ÷ 60

مثال تطبيقي:
الراتب:     8,000 ريال
أيام العمل: 22 يوم
ساعات العمل: 8 ساعات
═══════════════════════
تكلفة الدقيقة = 8000 ÷ 22 ÷ 8 ÷ 60 = 0.7576 ريال/دقيقة

إذا تأخر 15 دقيقة:
الخسارة = 15 × 0.7576 = 11.36 ريال
```

**المكونات:**

| الجدول                              | الوظيفة                                                                                                                                                  |
| ----------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `users` (حقول مالية)           | `basic_salary`, `housing_allowance`, `transport_allowance`, `other_allowances`, `working_days_per_month`, `working_hours_per_day`                   |
| `attendance_logs` (حقول مالية) | `cost_per_minute`, `delay_cost`, `early_leave_cost`, `overtime_value` — **لقطة فورية** من راتب الموظف وقت التسجيل |
| `financial_reports`                     | تقارير مالية شاملة بنطاقات متعددة (موظف / فرع / قسم / شركة) مع نسبة الخسارة                             |

**النقطة الحرجة:**

> كل سجل حضور **ينسخ تكلفة الدقيقة** من الموظف في لحظة التسجيل. هذا يعني أنه حتى لو تغير الراتب لاحقاً، تبقى البيانات التاريخية دقيقة 100%.

### 2.3 نظام الأدوار والصلاحيات — RBAC (10 مستويات)

**ماذا يفعل للمدير؟**

- يضمن أن **كل موظف يرى فقط ما يحق له**
- يحمي البيانات المالية الحساسة من الوصول غير المصرح
- يمنع **التعيين الجماعي** (Mass Assignment) للحقول الأمنية الحساسة

**الحقول المحمية (غير قابلة للتعيين الجماعي):**

- `is_super_admin` — لا يمكن ترقية أحد لمدير نظام إلا بالكود
- `security_level` — لا يمكن تغيير المستوى الأمني إلا عبر `setSecurityLevel()`
- `is_trap_target` — لا يمكن تفعيل المصيدة إلا عبر `enableTrapMonitoring()`

**مستويات الأدوار العشرة:**

| المستوى | الدور            | الصلاحيات الرئيسية                              |
| -------------- | --------------------- | ---------------------------------------------------------------- |
| 1              | متدرب            | حضور شخصي فقط                                         |
| 2              | موظف              | حضور + مالية شخصية + إجازات + بلاغات   |
| 3              | موظف أول       | + تصدير بيانات                                        |
| 4              | قائد فريق     | + عرض حضور الفريق + اعتماد إجازات       |
| 5              | مشرف              | + إدخال يدوي + اعتماد حضور                    |
| 6              | مدير قسم       | + مالية الفرع + إنشاء موظفين + تعاميم |
| 7              | مدير فرع       | + حذف موظفين + بلاغات + مصائد                |
|                |                       |                                                                  |
|                |                       |                                                                  |
| 8              | مدير إقليمي | + كل الفروع + إدارة كاملة                      |
| 9              | مدير تنفيذي | + إدارة رواتب + سجل المراجعة                |
| 10             | مدير النظام | **كل الصلاحيات بلا استثناء**          |

### 2.4 نظام المصائد النفسية (Psychological Trap System)

**ماذا يفعل للمدير؟**

- يكشف الموظفين ذوي **نوايا الاحتيال** قبل أن يحدث الضرر الفعلي
- يعرض عناصر واجهة مزيفة (مثل زر "اعتماد تحويل مالي") ويراقب من ينقر عليها
- يوثق كل تفاعل مع بيانات كاملة: الموظف، العنصر، الصفحة، عنوان IP، الوقت

**مستويات الخطر:**

| المستوى | الوصف                                        |
| -------------- | ------------------------------------------------- |
| `low`        | نقر استكشافي — ربما بالخطأ  |
| `medium`     | تفاعل مشبوه يستحق المراقبة |
| `high`       | محاولة واضحة للاستغلال        |
| `critical`   | خطر أمني فوري يتطلب تحقيق    |

---

### 2.5 البلاغات السرية (Secret Whistleblowing)

**ماذا يفعل للمدير؟**

- يوفر قناة **مشفرة 100%** للموظفين للإبلاغ عن المخالفات
- **لا يوجد ربط مباشر** بحساب المبلغ — مجهولية كاملة
- يتم التتبع عبر **رمز تذكرة مجهول** (`ticket_number`) و**رمز مُشفّر** (`anonymous_token`)
- المحتوى مشفر بـ **AES-256** عبر `encrypt()` / `decrypt()` في Laravel

---

### 2.6 المراسلات الذكية (Smart Messaging)

**ماذا يفعل للمدير؟**

- **محادثات مباشرة وجماعية** بين الموظفين
- **تعاميم رسمية** مع إمكانية طلب **إقرار بالاطلاع**
- **تنبيهات أداء آلية** تُنشأ تلقائياً عند تجاوز حدود معينة
- يعرف المدير من قرأ التعميم ومن لم يقرأ بعد

---

### 2.7 التلعيب (Gamification)

**ماذا يفعل للمدير؟**

- يحول **الانضباط** إلى **منافسة إيجابية**
- كل موظف يكسب **نقاط** على الحضور المبكر وعدم التأخير
- **شارات** تُمنح تلقائياً عند تحقيق إنجازات (مثل "30 يوم بدون تأخير")
- **سلسلة الأيام المتتالية** (Streak) تشجع على الاستمرارية
- مرتبط بالنظام المالي: شارة "صفر خسائر" = مكافأة 300 نقطة

**الشارات الحالية:**

| الشارة                | الشرط                                      | النقاط |
| --------------------------- | ----------------------------------------------- | ------------ |
| الطائر المبكر   | 5 أيام متتالية حضور مبكر     | 50           |
| سيد الانضباط     | شهر كامل بدون تأخير             | 200          |
| سلسلة حديدية (7) | 7 أيام متتالية                       | 70           |
| سلسلة ذهبية (30)  | 30 يوم متتالي                          | 500          |

> **ملاحظة v3.4:** الشارات الآن تُمنح عبر نموذج `UserBadge` المستقل — كل منحة تُسجَّل فيها: من منحها (`awarded_by`)، سبب المنح (`awarded_reason`)، ووقت المنح (`awarded_at`). النقاط تُضاف تلقائياً عبر `UserBadge::award()`.
| صفر خسائر           | شهر بدون خسارة مالية           | 300          |
| موفر التكاليف   | 50+ ساعة عمل إضافي                  | 150          |
| الأداء المتميز | أفضل موظف في الفرع               | 500          |
| بطل النزاهة       | اجتاز كل اختبارات المصائد | 1000         |

---

### 2.8 سجل المراجعة (Audit Trail)

**ماذا يفعل للمدير؟**

- يسجل **كل عملية** في النظام: إنشاء، تعديل، حذف، دخول، تصدير
- يحفظ **القيم القديمة والجديدة** لأي تعديل
- يربط كل عملية بالمستخدم وعنوان IP والمتصفح

## 4. المرحلة الأولى: منظومة الحضور والحدود الجغرافية (Phase 1)

### 4.1 آلية تسجيل الحضور الذكي

**كيف يعمل من جهة الموظف (PWA):**

1. يفتح الموظف تطبيق الويب ويضغط **"تسجيل حضور"**
2. المتصفح يطلب إذن GPS ويرسل الإحداثيات للخادم
3. النظام يحسب المسافة بين الموظف ومركز الفرع بصيغة **Haversine**
4. إذا كانت المسافة ≤ **17 متراً** → يُقبل التسجيل
5. إذا كانت المسافة > 17 متراً → يُرفض التسجيل مع رسالة واضحة

**كيف يراها المدير في لوحة القيادة (Filament):**

- جدول يومي لحضور كل الموظفين مع ألوان:
  - 🟢 **أخضر** = حاضر في الوقت (داخل الحدود)
  - 🟡 **أصفر** = متأخر (مع عرض دقائق التأخير وتكلفتها)
  - 🔴 **أحمر** = غائب
  - ⚠️ **تحذير** = حاضر لكن **خارج الحدود الجغرافية**
- عمود **"الخسارة المالية"** يعرض تكلفة التأخير بالريال لكل موظف
- فلاتر: حسب التاريخ، الفرع، القسم، الحالة
- أزرار تصدير CSV/Excel

### 4.2 شرح تكلفة التأخير للمدير

```
مثال عملي:
═══════════════════════════════════════════════════
الموظف: أحمد محمد
الراتب الأساسي: 8,000 ريال
أيام العمل: 22 يوم | ساعات العمل: 8 ساعات

تكلفة الدقيقة = 8,000 ÷ 22 ÷ 8 ÷ 60 = 0.7576 ريال

تأخر اليوم: 23 دقيقة
الخسارة = 23 × 0.7576 = 17.42 ريال
═══════════════════════════════════════════════════
```

> **النقطة الحاسمة:** النظام يحفظ **لقطة فورية** من تكلفة الدقيقة لحظة تسجيل الحضور.
> حتى لو تغير الراتب لاحقاً، البيانات التاريخية تبقى دقيقة 100%.

### 4.3 الخدمات المنفذة

| الخدمة             | الوظيفة                                                                                               |
| ------------------------ | ------------------------------------------------------------------------------------------------------------ |
| `GeofencingService`    | حساب المسافة بصيغة Haversine + التحقق من وجود الموظف داخل الحدود |
| `AttendanceService`    | تسجيل الحضور/الانصراف مع اللقطة المالية والتقييم الآلي        |
| `AttendanceResource`   | واجهة Filament للمدير لعرض وإدارة سجلات الحضور                               |
| `AttendanceController` | نقطة اتصال PWA لتسجيل الحضور بالـ GPS                                               |

## 5. المرحلة الثانية: نظام المصائد النفسية ومحرك المخاطر اللوغاريتمي (Phase 2)

### 5.1 استراتيجية الخداع الذكي

**الفلسفة:** بدلاً من منع السلوك المشبوه فقط، النظام **يكتشف النوايا** عبر عناصر واجهة مزيفة تبدو حقيقية.

**كيف يعمل للمدير:**

1. يُعرض للموظف المُراقب عناصر مزيفة تبدو كأنها حقيقية:
   - 🔍 **SALARY_PEEK** — زر "عرض رواتب الزملاء" → يعرض بيانات وهمية
   - 🔓 **PRIVILEGE_ESCALATION** — رابط "لوحة المدير" → يعرض رسالة نجاح مزيفة
   - ⏸️ **SYSTEM_BYPASS** — خيار "تعطيل الحضور" → يعرض تأكيد كاذب
   - 📤 **DATA_EXPORT** — زر "تصدير كل السجلات" → شريط تقدم وهمي + ملف CSV مشفر فارغ
2. **كل نقرة** تُسجّل في جدول `trap_interactions` مع بيانات كاملة
3. **النقاط التراكمية لوغاريتمية** — المرة الأولى = 10 نقاط، الثانية = 30، الثالثة = 70...

### 5.2 التسجيل اللوغاريتمي للمخاطر

```
المعادلة: نقاط_جديدة = 10 × (2^n - 1)
حيث n = عدد المرات التي تفاعل فيها الموظف مع المصائد

التصاعد:
  المرة 1 → 10 × (2¹ - 1) = 10 نقطة
  المرة 2 → 10 × (2² - 1) = 30 نقطة
  المرة 3 → 10 × (2³ - 1) = 70 نقطة
  المرة 4 → 10 × (2⁴ - 1) = 150 نقطة
  المرة 5 → 10 × (2⁵ - 1) = 310 نقطة

النتيجة: الموظف الذي يتكرر تفاعله يحصل على نقاط متسارعة بشكل حاد
```

### 5.3 واجهة المدير (Filament)

| المكون                | الوظيفة                                                                                                                               |
| --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| `TrapResource`            | إدارة تكوين المصائد (الاسم، الكود، الوزن، نوع الاستجابة)                                      |
| `TrapInteractionResource` | مراقبة حية لكل التفاعلات مع فلاتر وبحث                                                                       |
| `RiskWidget`              | أداة لوحة القيادة تعرض**"أعلى الموظفين خطورة"** مرتبة بالنقاط اللوغاريتمية |

### 5.4 أنماط الاستجابة المزيفة

| المصيدة       | نوع الاستجابة | ما يراه الموظف                                                         |
| -------------------- | ------------------------- | ---------------------------------------------------------------------------------- |
| SALARY_PEEK          | `success`               | جدول رواتب وهمي بأرقام عشوائية                           |
| PRIVILEGE_ESCALATION | `success`               | رسالة "تم منحك صلاحيات مؤقتة"                               |
| SYSTEM_BYPASS        | `warning`               | رسالة "تم تعطيل النظام مؤقتاً"                             |
| DATA_EXPORT          | `success`               | شريط تقدم → "اكتمل التصدير" → رابط CSV مشفر فارغ |

## 6. المرحلة الثالثة: واجهة الموظف (PWA) والبلاغات السرية والمراسلات الذكية (Phase 3)

### 6.1 بوابة الموظف (Employee PWA)

**الفلسفة:** واجهة "بدون جهد" — شاشة واحدة تعرض كل ما يحتاجه الموظف.

**رحلة المستخدم:**

1. يدخل الموظف ← تظهر لوحة القيادة فوراً بالعربية (RTL)
2. يرى 4 أدوات ذكية:
   - 🛰️ **أداة الحضور** — حالة GPS مباشرة (داخل/خارج نطاق 17م)
   - 🏆 **مركز النقاط** — النقاط والأيام المتتالية والشارات
   - 💰 **الأثر المالي** — "نقاط الانضباط" (تكلفة التأخير)
   - 📢 **التعميمات** — آخر التعميمات مع زر "قرأت واطلعت"
3. في الشريط الجانبي: المراسلات، البلاغات السرية
4. **المصائد مدمجة** بسلاسة — تبدو كجزء طبيعي من الواجهة

**التصميم:**

- خط **Tajawal** كخط أساسي
- RTL افتراضي مع دعم LTR للإنجليزية
- متجاوب (Mobile-First) عبر Tailwind CSS
- ألوان مؤسسية: أخضر زمردي + رمادي فحمي

### 6.2 نظام البلاغات السرية (القناة الآمنة)

**آلية السرية:**

- لا يُطلب اسم الموظف ولا بريده — بلاغ مجهول 100%
- يُنشأ **رمز تتبع** فريد (WB-XXXXXXXX-NNNNNN) + **رمز سري** للمتابعة
- المحتوى يُشفّر بـ `AES-256-CBC` عبر `encrypt()` قبل التخزين
- **لا يمكن** لأي شخص قراءة البلاغ إلا مستوى أمني 10 (المدير العام)

**تجربة المُبلّغ:**

1. يختار الفئة (fraud, harassment, corruption, safety)
2. يكتب تفاصيل البلاغ
3. يختار مستوى الخطورة
4. يحصل على رمز التتبع + الرمز السري
5. يمكنه لاحقاً الاستعلام بالرمز السري لمتابعة حالة البلاغ

### 6.3 المراسلات الذكية

| الميزة                        | الوصف                                                                       |
| ----------------------------------- | -------------------------------------------------------------------------------- |
| المحادثات المباشرة | رسائل بين موظفَين مع تأكيد القراءة                  |
| مجموعات العمل           | محادثات جماعية لفرق العمل                                  |
| التعميمات الرسمية   | إعلانات من الإدارة مع "إشعار القراءة"              |
| تكامل المصائد           | أزرار مزيفة (كـ "تسريب الرواتب") مدمجة بسلاسة |

## 7. غرفة القيادة: فلسفة "صفر جهد" (v1.4.0)

### 7.1 المفهوم

لوحة تحكم "غرفة القيادة" مصممة لتمنح المدير والمستوى 10 (Super Admin) رؤية فورية شاملة لصحة المنظمة بدون أي نقرات إضافية:

| المكوّن                         | الوظيفة                                                                                                         | التحديث                       |
| -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- | ------------------------------------ |
| عداد الخسائر اللحظي   | إجمالي الخسارة المالية من التأخيرات — جميع الفروع — اليوم الحالي | لحظي عند تحميل Filament  |
| خريطة أداء الفروع       | مقارنة بصرية بين الفروع: دقة الحضور + الامتثال للسياج الجغرافي      | بيانات الشهر الحالي |
| مركز تنبيهات النزاهة | آخر تفاعلات المصائد + حالات بلاغات المبلّغين (مستوى 10 حصرياً)         | لحظي                             |

### 7.2 محرك التحليل المالي

| التقرير                         | المعادلة                                                                         |
| -------------------------------------- | ---------------------------------------------------------------------------------------- |
| تحليل أثر التأخير (ROI) | `ROI = (potential_loss - actual_loss) / potential_loss × 100`                         |
| التنبؤ الشهري              | `predicted_monthly_loss = avg_daily_loss × remaining_working_days + accumulated_loss` |
| نطاق التقارير              | موظف فرد / قسم / فرع / الشركة بالكامل                          |

### 7.3 قبو البلاغات والمصائد (مستوى 10)

- فك تشفير بلاغات المبلّغين مع سجل المراجعة الكامل
- رسم بياني لنمو المخاطر بناءً على محرك المخاطر اللوغاريتمي
- بوابة أمان: `security_level >= 10` فقط — أي محاولة وصول غير مصرح بها تُسجَّل

## 12. إعادة الهيكلة المعمارية — نماذج الكيانات المستقلة (الإصدار 3.4)

### 12.1 المشكلة: انفصام معماري

كان النظام يستخدم 3 فلسفات مختلفة لجداول العلاقات:
1. **Pivot بدون موديل** — لجداول FK فقط (مثل `role_permission`)
2. **Pivot مع `withPivot()`** — لحقول بسيطة (مثل `conversation_participants`)
3. **Pivot بمنطق أعمال** — `user_shifts` و`user_badges` بهما تواريخ وحالات وشروط

المشكلة: الفلسفة الثالثة تخلق "كيان مُقنّع بزي Pivot". تعيينات المناوبات لها **صلاحية زمنية** (effective_from/to)، و**حالة** (is_current)، ومنح الشارات لها **توثيق** (awarded_at, awarded_reason). هذه ليست علاقات بسيطة — إنها عقود وإنجازات.

### 12.2 الحل: نماذج كيانات مستقلة

| النموذج الجديد | الجدول | الغرض |
|---------------|--------|-------|
| `UserShift` | `user_shifts` | عقد تعيين مناوبة — صلاحية زمنية، تدقيق، موافقة |
| `UserBadge` | `user_badges` | إنجاز موثق — مانح، سبب، وقت |

### 12.3 نموذج تعيين المناوبات (`UserShift`)

**ماذا يفعل للمدير؟**
- يوثق **من عيّن** هذا الشفت (assigned_by) و**من وافق** (approved_by)
- يحفظ **الفترة الزمنية** (من/إلى) مع إمكانية الإنهاء (`terminate()`)
- يمنع التعارض عبر `makeCurrent()` الذي يُلغي جميع التعيينات الأخرى تلقائياً
- يوفر **تاريخ كامل** للمناوبات عبر `shiftHistory()`

**الأعمدة:**

| العمود | النوع | الوصف |
|--------|------|-------|
| `user_id` | FK | الموظف المعيّن |
| `shift_id` | FK | المناوبة |
| `assigned_by` | FK nullable | من قام بالتعيين |
| `approved_by` | FK nullable | من وافق على التعيين |
| `effective_from` | date | بداية الصلاحية |
| `effective_to` | date nullable | نهاية الصلاحية (null = مفتوح) |
| `is_current` | boolean | هل هذا التعيين الحالي |
| `reason` | text nullable | سبب التعيين/الإنهاء |
| `approved_at` | timestamp nullable | وقت الموافقة |

### 12.4 نموذج منح الشارات (`UserBadge`)

**ماذا يفعل للمدير؟**
- يوثق **من منح** الشارة (awarded_by) — لا شارة بدون مسؤول
- يحسب **النقاط تلقائياً** عبر `UserBadge::award()` — يضيف نقاط للموظف وينشئ PointsTransaction
- يوفر scopes للاستعلام حسب الفترة (`awardedBetween`) أو الموظف (`forUser`)

### 12.5 تحويل العلاقات

| النموذج | العلاقة القديمة | العلاقة الجديدة |
|---------|----------------|----------------|
| `User` | `badges(): BelongsToMany(Badge)` | `badges(): HasMany(UserBadge)` |
| `User` | `shifts(): BelongsToMany(Shift)` | `shifts(): HasMany(UserShift)` |
| `User` | `currentShift(): ?Shift` (via pivot) | `currentShift(): ?Shift` (via UserShift→shift) |
| `Shift` | `users(): BelongsToMany(User)` | `assignments(): HasMany(UserShift)` |
| `Badge` | `users(): BelongsToMany(User)` | `awards(): HasMany(UserBadge)` |

**دوال جديدة على User:**
- `activeShift(): ?UserShift` — التعيين النشط الحالي
- `shiftHistory(): HasMany` — تاريخ المناوبات مرتب من الأحدث
- `awardedBadges(): HasMany` — الشارات مع eager-load لبيانات الشارة

### 12.6 مصحح البيانات (FixUserShiftsDataSeeder)

- يملأ الأعمدة الجديدة (`assigned_by`, `approved_by`, `approved_at`, `reason`, `awarded_by`) في السجلات الموجودة
- يُسند القيم للمدير الأعلى (Super Admin) افتراضياً مع سبب "ترحيل تلقائي"
- يُشغّل مرة واحدة فقط بعد تشغيل المايجريشن

### 12.7 مصانع الاختبار والاختبارات

| الملف | الغرض |
|-------|-------|
| `ShiftFactory` | مناوبات بأوقات واقعية (صباحي/مسائي/ليلي) مع حالات `inactive()` و`overnight()` |
| `BadgeFactory` | شارات بفئات وألوان ونقاط مع حالات `inactive()` و`withPoints()` |
| `UserShiftFactory` | تعيينات مع حالات `expired()`, `current()`, `withApproval()` |
| `UserBadgeFactory` | منح مع حالات `withReason()`, `awardedBy()` |
| `UserShiftTest` | 11 اختبار: علاقات، activeShift، currentShift (backward compat)، terminate، makeCurrent، isValidOn، scopes |

**ماذا يفعل للمدير؟**
- الموظف يرى **سجلاته فقط**
- المدير المباشر يرى سجلات فريقه
- مستوى 6+ يرى سجلات فرعه الكامل

### 13.3 التسجيل غير المتزامن — طابور الحضور (Queue)

**ماذا يفعل للمدير؟**

- **مسار جديد:** `POST /attendance/queue-check-in`
- المسار القديم لا يزال يعمل (متزامن) — **بدون كسر أي شيء**

### 13.4 إرسال التعاميم عبر الطابور (SendCircularJob)

**ماذا يفعل للمدير؟**
- إرسال تعميم لمئات الموظفين **بدون تجميد الصفحة**
- يُقسم الموظفين إلى دفعات (100) مع فاصل زمني لحماية قاعدة البيانات
- كل موظف يحصل على **تنبيه أداء** داخلي بمحتوى التعميم

### 13.5 الأحداث الذكية (Events & Listeners)

**ماذا يفعل للمدير؟**

| الحدث | ماذا يحدث تلقائياً؟ |
|-------|------------------------|
| **منح شارة** | تنبيه "تهانينا!" يظهر للموظف مع تفاصيل الشارة والنقاط |
| **تفاعل مع مصيدة** | تسجيل تلقائي في سجل التدقيق مع كل التفاصيل |
| **تسجيل حضور** | بوابة لربط أنظمة مستقبلية (إشعارات، تحليلات) |

### 13.6 معالجة الأخطاء الموحدة (BusinessException)

**ماذا يفعل للمدير؟**
- رسائل خطأ **واضحة بالعربية** بدلاً من أخطاء تقنية غامضة
- في الإنتاج: لا تظهر تفاصيل تقنية للمستخدم (حماية أمنية)
- كل خطأ يُسجل تلقائياً في سجل التدقيق (AuditLog)

### 13.7 التنبؤ بمغادرة الموظفين (ChurnPredictor)

**ماذا يفعل للمدير؟**
- يُحلل أنماط الحضور (آخر 30 يوم) ويحسب **درجة خطر المغادرة**
- 4 مستويات: منخفض — متوسط — مرتفع — حرج
- يُعطي **توصيات** بالعربية والإنجليزية لكل موظف
- **ملاحظة:** هذا نموذج أولي بقواعد ثابتة، سيُربط بنموذج ML فعلي في v5.0

### 13.8 لوحة التنبيهات (PerformanceAlertResource)

**ماذا يفعل للمدير؟**
- يعرض كل التنبيهات: شارات، تعاميم، تحذيرات تأخر، تحذيرات غياب
- **badge** يُظهر عدد التنبيهات غير المقروءة في القائمة الجانبية
- المدير يرى **تنبيهات فرعه فقط** (المدير العام يرى الكل)
- إمكانية **تحديد كمقروء** فردياً أو جماعياً

### 13.9 توثيق API التلقائي (Scramble)

**ماذا يفعل للمدير؟**
